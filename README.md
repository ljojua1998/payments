# Payments — გადახდების შედარების (Reconciliation) დეშბორდი

[![CI](https://github.com/ljojua1998/payments/actions/workflows/ci.yml/badge.svg)](https://github.com/ljojua1998/payments/actions/workflows/ci.yml)

ინსტრუმენტი კომპანიისთვის, რომელიც მართავს მომსახურების ხელშეკრულებებს.
საბანკო ტრანზაქციები ყოველდღიურად შემოდის საქართველოს ბანკის API-დან და
საჭიროა მათი შესაბამისობაში მოყვანა არსებულ ხელშეკრულებებთან — **ვინ გადაიხადა,
ვინ დააკლო და ვინ საერთოდ არ გამოჩენილა**.

**Live demo:** https://payments-orpin.vercel.app

## 📋 ამოცანა

დავალების არსი: ავიღოთ ორი დამოუკიდებელი მონაცემთა ნაკადი და შევკრათ ერთმანეთთან —

- **საბანკო ტრანზაქციები** — რა **გადაიხადეს** რეალურად (ფაქტობრივი)

   საბანკო API / CSV იმპორტი                ხელშეკრულებები (ბუღალტერი ავსებს)

### სასტარტო მონაცემები
- **15 კომპანია**, **18 ხელშეკრულება** (ზოგი შეჩერებული/დასრულებული)

## ✅ ძირითადი მოთხოვნები (დავალების scope)

- **ავტო-მატჩინგი** — ს/კ-ით (`sender_inn = company.tax_id`), `inn_exact`, confidence 1.00
- **სტატისტიკის ზოლი** — სულ / დამთხვეული / შეუსაბამო (რაოდენობა + თანხა) + match rate
- **ტრანზაქციების ცხრილი** — თარიღი, გამგზავნი, ს/კ, თანხა, ფერადი სტატუსი, კომპანია, მოქმედება; სორტი (თარიღი/თანხა), ფილტრი (სტატუსი)
- **თვეების ნავიგაცია** — ყველაფერი (stats, ცხრილი, შეჯამება) არჩეულ თვეს ასახავს
- **მოსალოდნელი vs ფაქტობრივი** — კომპანიების ჭრილში, ფერებით (მწვანე ≥ / წითელი < / ნაცრისფერი 0), მხოლოდ არჩეულ თვეში აქტიური ხელშეკრულებებით
- **Zod ვალიდაცია** — ფილტრები, ფორმები, API შემავალი
- **TanStack Query** — query key factory, cache invalidation mutations-ის შემდეგ, loading/error states, **optimistic updates**

## 🎯 სად დევს matching logic და რატომ

მატჩინგი **მონაცემთა ბაზაშია** — Postgres ფუნქცია
[`match_transactions_by_inn()`](./supabase/migrations/00005_matching_and_summary.sql),
რომელიც კლიენტიდან Supabase RPC-ით გამოიძახება („ავტო-მატჩინგი" ღილაკი).

```sql
UPDATE bank_transactions bt
SET matched_company_id = c.id, match_method = 'inn_exact',
    match_confidence = 1.00, status = 'matched'
FROM companies c
WHERE bt.sender_inn = c.tax_id AND bt.status = 'unmatched';
```

**რატომ ბაზაში და არა client-side:**
- **ერთი set-based UPDATE** ერთი round-trip-ით — client-side ვარიანტი ყველა
  ტრანზაქციის წამოღებას და თითო-თითო განახლებას მოითხოვდა (N+1)
- **ატომურობა** — ან ყველა დაემთხვევა, ან არცერთი; პარალელური გაშვება race
  condition-ს არ ქმნის
- **მასშტაბი** — 89 ტრანზაქციაზეც და 100 000-ზეც ერთნაირად, ინდექსით `sender_inn`-ზე
- ეხება **მხოლოდ** `status = 'unmatched'` რიგებს — ხელით მიბმული ან იგნორირებული
  ხელუხლებელი რჩება
- სახელის ვარიაცია პრობლემა არაა: შედარება მხოლოდ ს/კ-ით ხდება

„მოსალოდნელი vs ფაქტობრივი" შეჯამებაც ბაზაშია (`monthly_company_summary(month)`) —
აგრეგაცია მონაცემებთან ახლოს კეთდება, კლიენტამდე მხოლოდ 15 რიგი მიდის.

**ხელშეკრულება „აქტიურია არჩეულ თვეში"** თუ `start_date <= თვის ბოლო` და
(`end_date IS NULL` ან `end_date >= თვის პირველი`). ასე სეიფ ტრანსპორტი (შეჩერდა
15 მაისს) მაისში ჯერ ითვლება, ივნისში — აღარ; ურბან მუვერსი (დასრულდა 30 აპრილს)
მხოლოდ აპრილში ითვლება.
