## 🎯 სად დევს matching logic და რატომ

### როგორ მუშაობს მატჩინგი — ნაბიჯ-ნაბიჯ

საბანკო ტრანზაქცია და ხელშეკრულება ორ დამოუკიდებელ ნაკადში მოდის. ერთადერთი
საიმედო გასაღები, რომლითაც ისინი ერთდება, არის **საიდენტიფიკაციო კოდი (ს/კ)** —
გამგზავნის სახელი შეიძლება სხვადასხვანაირად ეწეროს („შპს გეოტრანსი",
„გეოტრანსი", „გეოტრანსი (ფილიალი)"), მაგრამ ს/კ ერთი და იგივეა.

1. **ავტო-მატჩინგი** — მომხმარებელი აჭერს „ავტო-მატჩინგი" ღილაკს, კლიენტი კი
   Supabase RPC-ით იძახებს ბაზის ფუნქციას
   [`match_transactions_by_inn()`](./supabase/migrations/00005_matching_and_summary.sql):

   ```sql
   UPDATE bank_transactions bt
   SET matched_company_id = c.id, match_method = 'inn_exact',
       match_confidence = 1.00, status = 'matched'
   FROM companies c
   WHERE bt.sender_inn = c.tax_id      -- ზუსტი დამთხვევა ს/კ-ით
     AND bt.status = 'unmatched';      -- მხოლოდ ჯერ დაუმთხვეველი რიგები
   ```

   ერთი set-based `UPDATE` ყველა `unmatched` ტრანზაქციას, რომლის `sender_inn`
   ემთხვევა კომპანიის `tax_id`-ს, მიაბამს ამ კომპანიას, უსვამს `status = 'matched'`,
   `match_method = 'inn_exact'` და `confidence = 1.00`.

2. **შეუსაბამო რჩება unmatched** — ტრანზაქცია, რომლის ს/კ არც ერთ კომპანიას არ
   ემთხვევა (seed-ში 12 ასეთი „უცნობი გამგზავნია"), `unmatched`-ად რჩება და
   წითლად აღინიშნება.

3. **ხელით დამუშავება** — შეუსაბამო ტრანზაქციაზე მომხმარებელს შეუძლია კომპანია
   ხელით მიაბას (`manual`), მოხსნას მიბმა, ან დააიგნოროს. მიბმა/მოხსნა
   optimistic update-ია — მყისიერად აისახება UI-ში, შეცდომისას rollback ხდება.

4. **Fuzzy შეთავაზება (ბონუსი)** — შეუსაბამო რიგზე `pg_trgm`-ით და სახელის
   ნორმალიზაციით (შპს/სს პრეფიქსისა და ფრჩხილების მოცილება) სისტემა თავად
   თავაზობს სავარაუდო კომპანიას (მაგ. „გეოტრანსი (ფილიალი)" → „შპს გეოტრანსი"),
   ერთი კლიკით მისაბმელს.

**რატომ ბაზაში და არა client-side:**
- **ერთი set-based UPDATE** ერთი round-trip-ით — client-side ვარიანტი ყველა
  ტრანზაქციის წამოღებას და თითო-თითო განახლებას მოითხოვდა (N+1)
- **ატომურობა** — ან ყველა დაემთხვევა, ან არცერთი; პარალელური გაშვება race
  condition-ს არ ქმნის
- **მასშტაბი** — 89 ტრანზაქციაზეც და 100 000-ზეც ერთნაირად, ინდექსით `sender_inn`-ზე
- ეხება **მხოლოდ** `status = 'unmatched'` რიგებს — ხელით მიბმული ან იგნორირებული
  ხელუხლებელი რჩება

### მოსალოდნელი vs ფაქტობრივი

შეჯამებაც ბაზაშია (`monthly_company_summary(month)`) — აგრეგაცია მონაცემებთან
ახლოს კეთდება, კლიენტამდე მხოლოდ ~15 რიგი მიდის:

- **მოსალოდნელი** — არჩეულ თვეში მოქმედი ხელშეკრულებების `monthly_amount`-ების ჯამი
- **ფაქტობრივი** — ამ თვეში ამ კომპანიაზე **მიბმული** ტრანზაქციების ჯამი
- **სხვაობა** — ფაქტობრივი − მოსალოდნელი (მწვანე ≥ / წითელი < / ნაცრისფერი 0)

**ხელშეკრულება „მოქმედია არჩეულ თვეში"** თუ `start_date <= თვის ბოლო` და
(`end_date IS NULL` ან `end_date >= თვის პირველი`). ასე სეიფ ტრანსპორტი (შეჩერდა
15 მაისს) მაისში ჯერ ითვლება, ივნისში — აღარ; ურბან მუვერსი (დასრულდა 30 აპრილს)
მხოლოდ აპრილში ითვლება. ე.ი. „მოსალოდნელი" კონტექსტს თვის მიხედვით ცვლის.

**საკვანძო გადაწყვეტილებები:**
- **მდგომარეობა URL-შია** (`?month=2026-06&status=unmatched&q=...`) — Zod-ით
  ვალიდირდება, ლინკი გაზიარებადია, back/forward მუშაობს
- **Optimistic updates** — ხელით მიბმა/იგნორი მყისიერად აისახება, შეცდომისას rollback

  დოკუმენტები/ავატარები/პროფილები owner-only
- **AI კონტექსტი ცოცხალია** — ასისტენტი ყოველ კითხვაზე ბაზიდან კითხულობს, არაფერს იგონებს
- **Security headers** — HSTS, CSP, X-Frame-Options, X-Content-Type-Options,
  Referrer-Policy და Permissions-Policy ყველა პასუხზე (`next.config.ts`)
- **პაროლის დაცვა** — leaked-password protection Supabase-ის Pro გეგმის ფუნქციაა;
  free tier-ზე პაროლის ვალიდაცია აპლიკაციის დონეზე ხდება (Zod), OTP-ს ვადა 3600 წმ
